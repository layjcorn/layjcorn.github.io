Sensor-Actuated Valve System with Autonomous Fault Recovery

The core capability of this system is closed-loop jam detection. The primary function detects hand presence via an infrared QRD sensor, triggering a Parallax Propeller 2 microcontroller to drive a PZ22 DC gear motor through an inverted-logic relay module. Rather than relying on simple software timeouts, the system autonomously detects and recovers from mechanical valve jams by measuring motor current draw across a 5Ω shunt resistor passed through an LM358N op-amp. This moves the project from a basic open-loop actuator into a robust controls engineering implementation.

pub Jammer()
  repeat
    if ina.[4] == 0 or ina.[5] == 0                     ' only sense when a relay is on (otherwise it will read 0V)
      jam := pinread(1)
    else                                                ' set to 1 when motor is stopped
      jam := 1
      waitms(100)

' Inside Main() Loop:
      if jam == 0                                       ' if pin 1 reads 0 = shunt resistor detected jam
          waitms(500)                                   ' quick wait and double-check (was getting too many false positives)
          if jam == 0                                   ' double-check it's a real jam
            numjams++                                   ' keep track of # of jams
            if numjams == 1                               ' if first jam open up and try to close again
              pinhigh(5)                                  ' IN2 off (red) - stops CW rotation

              pinlow(4)                                   ' IN1 on (black) - starts CCW rotation

              pinhigh(4)                                  ' IN1 off (black) - stops rotation
              waitms(100)
              pinlow(5)                                   ' IN2 on (red) - starts CW rotation
              waitms(100)
      pinhigh(5)                                        ' IN2 off (red) - stops CW rotation
      repeat until hand == 0                            ' ensure hand is removed before next iteration
      waitms(100)
            if numjams == 2                               ' second jam stops everything

Design Notes and Future Work

While the 5Ω shunt resistor provided a reliable, fast-response method for jam detection, a production revision of this system would require integration with a physical tap fitting and the addition of an inline flow rate sensor to positively confirm successful dispensing, closing the fluidic loop alongside the electromechanical loop. Furthermore, a sealed, IP67-rated enclosure would be necessary to protect the control electronics from the operating environment.

Cam Profile Accuracy: Servo vs. Stepper Motor Under Proportional Control

This project isolates a specific engineering problem: which motor type more accurately reproduces a defined, non-linear cam profile, and by how much? By treating the motor type as the independent variable and profile tracking accuracy as the dependent variable, we evaluated both platforms under identical proportional control structures. The stepper motor ultimately produced significantly higher accuracy due to its fundamental step-resolution mechanics and dynamic microstepping capabilities.

Test Methodology

The target profile was a variable-radius cam designed to produce a non-harmonic stroke. Actual mechanical displacement was recorded using a linear rotary encoder on the follower mechanism. Error was quantified using Root Mean Square (RMS) deviation across continuous full-rotation cycles at a controlled duty cycle.

Key Finding: Stepper motor control produced 93% lower profile tracking error (RMS) than servo control, attributed to the stepper's dynamic microstepping and inherent physical position resolution advantage.

Kinematic Math & Microstepping Implementation

To accurately reproduce the piecewise, non-harmonic waveform, the target velocity (v) was calculated in real-time across four distinct mathematical bounds using dedicated Propeller 2 parallel processing (Cog 1). The base speed was dynamically converted into a microstepping WaitTime, actively adjusting mechanical resolution based on the curve's instantaneous slope.

pub Math() | y, v, base_speed, temp_val
  repeat
    y := float(Position) /. 3200.0    ' Get step #

    if direction == 1                 ' Going up
      if Position < 1440
        v := 3.14159 *. fsqrt(0.2025 -. (y *. y))

      elseif Position < 1760
        temp := 1.0 -. ((y -. 0.45) /. 0.05)
        v := 0.47124 *. fsqrt(1.0 -. (temp *. temp))

      else
        temp := 1.0 -. ((y -. 0.55) /. 0.45)
        v := 1.4137 *. fsqrt(1.0 -. (temp *. temp))
    else                              ' Going down
      temp := (2.0 *. y) -. 1.0
      v := 2.35619 *. fsqrt(1.0 -. (temp *. temp))

    if v < 0.1                       ' When v ~ 0 (smoothing)
      v := 0.1

    WaitTime := trunc(1500 /. v)      ' Truncate the # to ensure WaitTime isn't a decimal

Real-Time Multiplayer Input on a Propeller 2 Microcontroller

The Propeller 2 is a multi-core microcontroller featuring eight independent parallel processors. Moving beyond standard single-core embedded applications, this project implemented a "Whack-a-Mole" style game by distributing concurrent hardware input handling, game state logic, and VGA display rendering across separate physical cores. This architecture avoids standard interrupt-driven timing conflicts and introduces complex considerations for inter-core communication and shared memory access.

The primary hardware challenge was decoding Nintendo Wii controller inputs over an I2C-derived protocol. Because the Propeller 2 operates with deterministic timing, the input parsing routine had to strictly manage clock stretching and data frame extraction without blocking the primary game logic.

Work was divided efficiently: Core 0 was dedicated exclusively to generating the VGA timing signals, Core 1 maintained the authoritative game state and scoring, and Core 2 acted as an autonomous input driver, reading HID data and dropping sanitized player coordinates into a shared memory register.
